# Plan 08-01: Slave Status and Live Preview Firmware Protocol

**Phase:** 08 - Slave Monitor, Sync Evidence, and CSV Export
**Status:** Planned
**Wave:** 1

## Objective

Add a low-rate slave-to-master monitor protocol so the master knows which slaves are alive and whether each slave is sensor-responsive, SD-ready, recording, saving samples, synchronized, and producing quaternion channels.

## Tasks

1. Define a packed ESP-NOW `SlaveMonitorPacket` shared by master and slave sketches.
   - Include protocol magic/version, slave ID/MAC-derived identity, sample rate, sequence, online counters, SD state, recording state, saved samples, SD errors, IMU OK, mag OK, quaternion enabled, DIO level, DIO edge count, sync received, clock offset, and compact live values.
   - Live values should include accel, gyro, mag, quaternion, and DIO for selected-slave UI at about 5 Hz.

2. Update slave firmware to send monitor packets around 5 Hz.
   - Use ESP-NOW broadcast to avoid per-master MAC configuration.
   - Do not block acquisition or SD writer paths.
   - Keep packet generation lightweight and tolerate dropped packets.

3. Update master firmware to store recent slave monitor packets.
   - Key by MAC and optional slave ID.
   - Track `last_seen_ms` and mark status stale/offline after a short threshold.
   - Store a fixed maximum number of slots, initially at least 6.

4. Expose machine-readable master status lines.
   - Add `SLAVE_STATUS ...` lines to `STATUS` output.
   - Include compact aggregate counts: active slaves, SD-ready slaves, recording slaves, error count.
   - Add the same or equivalent summary to `REC STATUS` if practical so the plugin can poll it without a separate command.

5. Fix multi-slave IP identity.
   - Remove hard-coded identical slave static IP, or make it configurable by `SLAVE_ID`.
   - Preserve the default one-slave setup, but avoid IP conflicts when several slaves boot.

6. Keep current relay-only recording semantics.
   - Master without SD remains valid.
   - `REC START`/`REC STOP` broadcast commands still trigger all slaves.
   - Master status must not claim slave recording success unless recent slave monitor data supports it.

## Verification

- Master boot shows `ESP-NOW ready (role=master ch=6 iface=AP)`.
- Slave boot shows `ESP-NOW ready (role=slave ch=6 iface=STA)` and `SD ready`.
- Master `STATUS` shows at least one `SLAVE_STATUS` line within 2 seconds of slave boot.
- Moving/rotating slave changes live accel/gyro/quaternion fields in the status packet.
- Starting/stopping plugin RECORD updates slave `recording` and `saved_samples` fields.
- With slave powered off, master marks it stale/offline rather than showing old OK state.
- With two slaves powered, identities do not collide.

## Success Criteria

- Operator can query the master and tell whether slaves are online, SD-ready, sensor-responsive, recording, saving samples, synchronized, and error-free.
- Slave monitoring adds no SD write errors or acquisition loop overruns in a 1-minute smoke run.
