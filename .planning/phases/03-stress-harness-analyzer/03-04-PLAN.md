<objective>
Diagnose and fix the SD-on regression observed after moving ICM20948 to SPI.

The failure signature is not duplicate sensor samples: `dup=0` remains clean. The failure
is SD-on acquisition pressure: nonzero `sd_err`, high `max_sd_us`, gaps at some rates, and
loop overruns even at 500 Hz. The likely change from yesterday is that SD and ICM now share
the SPI bus from different tasks. This plan first makes the counters truthful, then tests a
shared SPI lock and batched SD writes so the final result explains why SD-on performance
changed and what safe rate remains.
</objective>

  <name>Task 1: Split SD error counters before changing behavior</name>
  <files>esp32/arduino/step_node/step_node.ino, esp32/host/stress_test_serial.py</files>
  <action>
    Replace the overloaded SD error reporting with separate counters so the next runs can
    identify the failure mode.

    Firmware counters:
    - `g_sd_queue_drops`: increment only when `xQueueSend(g_sd_queue, &rec, 0)` fails.
    - `g_sd_write_errors`: increment only when `g_sd_file.write(...)` returns a short write.
    - `g_sd_header_errors`: increment when the session header write fails.
    - `g_sd_open_errors`: increment when `SD.open(...)` fails.
    - `g_sd_mutex_timeouts`: increment when the SD writer cannot take its mutex.
    - `g_sd_flush_count`, `g_max_sd_flush_us`: track periodic and final flush cost separately.
    - `g_sd_queue_max_depth`: track `uxQueueMessagesWaiting(g_sd_queue)` maximum after enqueue.

    Preserve backward compatibility:
    - Keep `errors=` in `SD_FINAL` as the sum of queue drops + write/header/open/mutex errors.
    - Add explicit fields: `queue_drops=`, `write_errors=`, `header_errors=`,
      `open_errors=`, `mutex_timeouts=`, `max_queue_depth=`, `flush_count=`,
      `max_flush_us=`.

    Update `stress_test_serial.py` to parse and display the new fields in SUMMARY notes.
  </action>

<task type="auto">
  <name>Task 2: Add one shared SPI bus lock for ICM and SD operations</name>
  <files>esp32/arduino/step_node/step_node.ino</files>
  <action>
    Add a global SPI bus mutex used by every SPI peripheral path in this sketch.

    Required behavior:
    - Create `g_spi_mutex` during setup before ICM init and SD task startup.
    - Wrap all ICM SPI transactions (`icmReadBytes`, `icmWrite`, WHO_AM_I reads, aux
      read/write helpers through those functions) with the shared SPI mutex.
    - Wrap SD `write`, `flush`, `open`, `close`, and checksum read access with the same
      SPI mutex.
    - Keep critical sections short. Do not hold the SPI mutex while waiting on the SD queue
      or while doing non-SPI bookkeeping.
    - Count mutex acquisition failures separately using `g_sd_mutex_timeouts` or an
      explicit SPI timeout counter if a timeout is used.

    Use a helper pair rather than scattering raw semaphore calls:
    - `spiBusTake(TickType_t timeout)`
    - `spiBusGive()`

    The acquisition loop must not block indefinitely on SPI. If a bounded timeout is used
    for ICM reads, return fallback/cached data and count the failure.
  </action>

<task type="auto">
  <name>Task 3: Batch SD queue records into larger writes</name>
  <files>esp32/arduino/step_node/step_node.ino</files>
  <action>
    If split counters show queue buildup or high write overhead after the SPI lock, batch
    queued records in the SD task.

    Required behavior:
    - Drain up to a fixed batch size from `g_sd_queue` into a RAM buffer after the first
      blocking receive succeeds.
    - Write the contiguous batch with one `g_sd_file.write(...)` call.
    - Count saved samples by `bytes_written / sizeof(SdLogRecord)` only when the full batch
      writes successfully.
    - Preserve record format exactly; do not change `SdLogHeader`, `SdLogRecord`, or the
      analyzer contract.
    - Choose a conservative batch size that fits ESP32 RAM comfortably, for example 32 or
      64 records.
    - Keep flush cadence controlled by time and finalization, not every batch.

    Do not add more queue depth until this batching path has been tested.
  </action>
