<objective>
Make SD-on stress-test status collection deterministic enough that pass/fail means what it says.

Current high-rate runs can show impossible rows such as `sd_saved=None` or `sd_saved=159` while the live stream is clean and neighboring rates pass. The harness is accepting broad text status and timing out around firmware finalization. This plan adds a unique final receipt from firmware and makes the host wait for that exact receipt before evaluating SD-on runs.
</objective>

<task type="auto">
  <name>Task 2: Parse only the final SD receipt for SD-on pass/fail</name>
  <files>esp32/host/stress_test_serial.py</files>
  <action>
    Add a host helper that waits for a specific final receipt instead of accepting any `STATUS` or `SD_STATUS` line.

    Required implementation:
    - Add `SD_FINAL_RE = re.compile(r"SD_FINAL\\s+([^\\r\\n]*)")` near `STATUS_RE`.
    - Add `wait_for_sd_final(ser: serial.Serial, timeout: float) -> dict[str, str]`.
    - The helper should read serial bytes until it sees an `SD_FINAL` line or the timeout expires.
    - Parse key/value fields from only that `SD_FINAL` line.
    - Do not treat generic `STATUS`, `SD_STATUS`, `STOPPED`, or binary-frame text as a final receipt.
    - If no `SD_FINAL` arrives, return `{}`.

    Update `test_one_rate()` SD-on stop flow:
    - Keep `STOP`, `wait_for_text(... STOPPED ...)`, and `drain_serial(...)`.
    - Send `RECORD OFF`.
    - Replace the current broad `status_text = wait_for_text(...); status.update(parse_key_values(status_text))` with `status.update(wait_for_sd_final(ser, 20.0))`.
    - After final receipt parsing, still call `read_status(ser, 1.0)` as a secondary source for non-final counters, but final `saved/errors/max_sd_write_us/overrun` should come from `SD_FINAL` when present.

    Field mapping:
    - `sd_saved = first_int(status, "saved", "sd_saved")`
    - `sd_errors = first_int(status, "errors", "sd_errors")`
    - `max_sd_write_us = maybe_int(status, "max_sd_write_us")`
    - `loop_overruns = first_int(status, "overrun", "loop_overruns")`
  </action>

  <files>esp32/host/stress_test_serial.py</files>
  <action>
    Extend result reporting so a missing final SD receipt is distinguishable from a confirmed failure.

    Required implementation:
    - Add a `status` or `result` string field to `RateResult` with values `PASS`, `FAIL`, or `UNKNOWN`.
    - For non-SD runs, preserve current behavior: pass rule produces `PASS` or `FAIL`.
    - For SD-on runs, if `wait_for_sd_final(...)` returns no counters, set result to `UNKNOWN`, set `passed = False`, and set `note` to include `missing SD_FINAL`.
    - Do not allow an SD-on run to become `PASS` without a final receipt.
    - In stdout and SUMMARY output, show `UNKNOWN` instead of plain `FAIL` for missing final receipt.
    - Keep highest-passing Hz based only on confirmed `PASS` rows.
    - Recommended cap remains 80% of the highest confirmed pass.

    This change makes the harness honest:
    - clean stream + no final receipt = UNKNOWN
    - final receipt with errors/drops/overrun = FAIL
    - clean stream + clean final receipt = PASS
  </action>
