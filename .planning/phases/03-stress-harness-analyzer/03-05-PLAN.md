<objective>
Apply software tuning to reduce SPI bus contention between the shared ICM20948 and SD card
paths, based on the hardware results from 2026-06-17.

Three symptoms to address:
1. Loop overruns at 100 Hz — every sample polls mag (10 ms poll period = 100 Hz), so the
   contended loop is ICM-wait(5 ms) + mag-wait(5 ms) + VQF(1 ms) = 11 ms > 10 ms period.
2. ~90% mean Hz at 500–700 Hz — SD batch fires every 64 ms; 10-12 ms SD write latency
   causes the loop to run at ~91 Hz during writes instead of the target rate.
3. 770–780 Hz ceiling with filter+SD — same contention, at the filter-on upper boundary.

Two changes:
- Increase SD_WRITE_BATCH_RECORDS 32 → 128. Batches fire 4× less often (every 256 ms at
  500 Hz). Projected mean Hz at 500 Hz improves from ~90% to ~97%.
- Reduce ICM and mag SPI timeouts 5 ms → 2 ms. Contended loop becomes 2+2+1 = 5 ms.
  Fixes 100 Hz overruns (5 ms < 10 ms period). Does not fix 500+ Hz overruns during writes,
  but these are now shorter and 4× less frequent.

Hardware tests confirm the before state. After flashing, the same sweep proves the after state.
</objective>

<task type="auto">
  <name>Task 1: Increase SD write batch size 32 → 128</name>
  <files>esp32/arduino/step_node/step_node.ino</files>
  <action>
    Change the compile-time constant:

      #define SD_WRITE_BATCH_RECORDS 32
    →
      #define SD_WRITE_BATCH_RECORDS 128

    No other changes are needed. The sdWriteTask drains up to SD_WRITE_BATCH_RECORDS records
    from g_sd_queue into a stack-allocated batch array before writing. Increasing to 128:
    - RAM cost: 128 × sizeof(SdLogRecord) = 128 × 40 = 5,120 bytes on the sdWriteTask stack.
    - Increase sdWriteTask stack from 4096 → 8192 to accommodate the larger batch buffer.
    - Batch frequency at 500 Hz drops from every 64 ms to every 256 ms.
    - SPI bus is held for the same total time per second, but in 4× fewer, 4× longer bursts.
      Fewer bursts = fewer acquisition-loop interruptions.

    Locate the xTaskCreatePinnedToCore call for sdWriteTask and update stack size:

      xTaskCreatePinnedToCore(sdWriteTask, "sd_write", 4096, NULL, 4, NULL, 0);
    →
      xTaskCreatePinnedToCore(sdWriteTask, "sd_write", 8192, NULL, 4, NULL, 0);
  </action>
  <verify>
    <automated>
      arduino-cli compile --fqbn esp32:esp32:XIAO_ESP32S3 esp32\arduino\step_node
      rg -n "SD_WRITE_BATCH_RECORDS" esp32/arduino/step_node/step_node.ino
    </automated>
    <expected>SD_WRITE_BATCH_RECORDS 128 in output; compile clean.</expected>
  </verify>
  <done>SD batch size is 128; sdWriteTask stack is 8192 bytes.</done>
</task>

<task type="auto">
  <name>Task 2: Reduce ICM and mag SPI timeouts 5 ms → 2 ms</name>
  <files>esp32/arduino/step_node/step_node.ino</files>
  <action>
    Change all acquisition-path spiBusTake timeout calls from pdMS_TO_TICKS(5) to
    pdMS_TO_TICKS(2). These are the three ICM SPI helper functions:
    - icmWriteAddr
    - icmReadAddr
    - icmReadBytes

    Do NOT change the portMAX_DELAY calls used in sdWriteTask, session open/close,
    checksumSdFile, or record retrieval — those are outside the acquisition hot path and
    may legitimately block.

    Do NOT change the sdWriteTask's own spiBusTake(portMAX_DELAY) call.

    After this change:
    - Contended ICM read waits at most 2 ms instead of 5 ms.
    - Contended mag read (via icmReadBytes) waits at most 2 ms instead of 5 ms.
    - Contended loop at 100 Hz: 2 ms ICM + 2 ms mag + 1 ms VQF = 5 ms < 10 ms period.
    - No overruns at 100 Hz unless an SD write takes longer than 8 ms (extremely rare with
      batch=128 and a typical SD card).
    - At 200 Hz (5 ms period): mag polls every other sample; contended non-mag loop =
      2 ms + 1 ms = 3 ms < 5 ms; contended mag loop = 2+2+1 = 5 ms → borderline but
      batch=128 fires rarely enough to not dominate mean Hz.

    spi_mutex_timeouts already counted separately — no counter changes needed.
  </action>
  <verify>
    <automated>
      arduino-cli compile --fqbn esp32:esp32:XIAO_ESP32S3 esp32\arduino\step_node
      rg -n "pdMS_TO_TICKS" esp32/arduino/step_node/step_node.ino
    </automated>
    <expected>
      All acquisition-path spiBusTake calls show pdMS_TO_TICKS(2).
      portMAX_DELAY calls remain unchanged in sdWriteTask and session management.
      Compile clean.
    </expected>
  </verify>
  <done>ICM and mag SPI timeouts are 2 ms; portMAX_DELAY paths are unchanged.</done>
</task>
