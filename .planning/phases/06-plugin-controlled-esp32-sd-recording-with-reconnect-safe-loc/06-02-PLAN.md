# Plan 06-02: Firmware and Bridge Recording Control

**Phase:** 06 - Plugin-Controlled ESP32 SD Recording With Reconnect-Safe Local Retrieval
**Status:** Planned
**Wave:** 2
**Depends on:** 06-01
**Requirements:** SD-01, SD-02, SD-03, SD-04, SD-05, LOSS-01, LOSS-03, LOSS-04, STALL-01, STALL-02, STALL-03

## Scope

Implement firmware and bridge support so plugin RECORD commands actually start/stop ESP32 SD logging, survive temporary host disconnects, expose truthful status, and transfer finalized SD files without corrupting live sample transport. This plan must resolve the review feedback about firmware target ownership, failure-mode-specific disconnect handling, non-blocking recording/status behavior, serial retrieval isolation, and concrete transfer errors.

## Decision Trace

- D-01: Firmware implementation must keep normal ESP32 recording board-side on SD.
- D-04: Firmware must flush/close finalized SD recordings and expose retrieval for completed files.
- D-05: Firmware and bridge transfer must remain post-record export, not the primary live acquisition path.
- D-07: Firmware must continue SD logging through host/plugin/bridge connection loss.
- D-08: Supported firmware paths must define the reconnect grace period as an easy-to-change top-of-file constant.
- D-09: Firmware must continue the same recording session when reconnect happens within the grace period.
- D-10: Firmware must finalize on grace timeout and retain session metadata for the next host connection.
- D-11: Firmware and bridge commands must explicitly acknowledge record on/off, timeout finalization, and retrieval outcomes.
- D-12: Firmware and bridge status must expose SD, recording, counter, latency, error, finalization, and session metadata fields.
- D-13: Direct TCP and USB bridge paths must either behave consistently or fail truthfully when unsupported.

## Tasks

1. Make firmware targets explicit before editing:
   - ESP-IDF direct TCP path is canonical for `esp32/firmware/main/open_ephys_stream.c` plus `sd_logger.c/h`.
   - Arduino USB bridge path in `esp32/arduino/step_node/step_node.ino` must either implement the same operator-level protocol subset or be documented as unsupported with clear bridge/plugin failure behavior.
   - `esp32/host/serial_tcp_bridge.py` must relay the supported command/status/retrieval surface for bridge mode.
2. Add a top-of-file configurable reconnect grace constant to each supported firmware path, defaulting to a one-to-two-minute value.
3. Extend firmware recording state so SD logging can continue when the host command connection drops, while status queries during recording remain non-blocking for the acquisition loop.
4. Implement disconnect handling separately for direct TCP client disconnect, bridge/plugin socket disconnect, serial device loss, bridge process loss as observable by firmware/host, transfer disconnect, and firmware reboot recovery. Each path must preserve enough metadata for the next connection to report the session outcome.
5. On host disconnect during recording, start the grace timer and keep appending samples to SD.
6. On reconnect before timeout, cancel the grace timer and continue the existing recording session without a false stop/start boundary.
7. On timeout, flush/close the SD file, stop logging, record finalization reason as disconnect timeout, and preserve final status until explicit clear or new recording start.
8. Add or extend firmware status commands so direct TCP and serial paths can report protocol version/capabilities, recording state, transfer state, counters, finalization reason, file metadata, and retained finalized-session metadata.
9. Add file retrieval commands or a transfer endpoint that reads only finalized SD files by session id/token after recording stops. Retrieval must return explicit errors for busy_recording, not_finalized, not_found, offset_out_of_range, checksum_mismatch, sd_error, aborted, unsupported_protocol, and transfer_in_progress.
10. Define serial retrieval behavior explicitly: retrieval must run only after recording finalization, report expected duration/progress from byte counts, support abort/retry from offset, and must not block acquisition/status paths during live recording.
11. Update `esp32/host/serial_tcp_bridge.py` so USB bridge mode relays `RECORD`, capability/status, metadata, retrieval, abort, and clear-status commands, not just `FREQ`, `CFG`, `FILTER`, and `STOP`.
12. Ensure file-transfer framing cannot be mistaken for live Open Ephys sample frames by implementing the separation rule selected in 06-01.
13. Preserve SD source files after transfer attempts. Retrieval is read-only and never deletes/truncates the finalized SD source, even after successful local copy.

## Verification

- Code inspection confirms SD logging continues after host disconnect while the grace timer is active.
- Reconnect-before-timeout test confirms the session continues and `sd_saved` keeps increasing.
- Timeout test confirms the file is flushed/closed and status reports timeout finalization.
- USB bridge test confirms `RECORD ON/OFF` reaches Arduino firmware and status returns.
- Retrieval test confirms a finalized SD file can be copied from ESP32 to local storage without corrupting the SD source file.
- SD continuity test proves recording continues while the host/plugin/bridge is gone using the SD analyzer, not status counters alone.
- Direct TCP client disconnect, plugin-to-bridge disconnect, bridge process loss, USB/serial loss, transfer disconnect, and firmware reboot behavior are either tested or explicitly marked unsupported with truthful status.
- Retrieval during active recording returns `busy_recording` or equivalent and does not disturb SD logging.
- Transfer error paths return the contract-defined errors for missing/not-finalized files, bad offsets, checksum mismatch, SD errors, abort, and unsupported protocol.
- ESP-IDF firmware build is run where the local environment supports it; otherwise the plan summary records the missing toolchain. Python bridge changes pass syntax/static validation.
- `git diff --check` passes for changed files.
