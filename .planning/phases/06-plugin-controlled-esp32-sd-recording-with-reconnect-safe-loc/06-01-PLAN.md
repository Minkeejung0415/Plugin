# Plan 06-01: Recording Protocol and Status Contract

**Phase:** 06 - Plugin-Controlled ESP32 SD Recording With Reconnect-Safe Local Retrieval
**Status:** Planned
**Wave:** 1
**Requirements:** SD-01, SD-03, SD-04, SD-05, LOSS-03, LOSS-04, STALL-03, OPS-01

## Scope

Define the command/status/file-transfer contract shared by the plugin, USB bridge, direct TCP path, and firmware before implementing transport-specific code.

## Tasks

1. Document the RECORD lifecycle states: idle, starting, recording, host-disconnected-grace, finalizing, finalized, retrieval-in-progress, retrieved, failed.
2. Define command names and responses for record start, status query, record stop/finalize, file metadata query, file chunk retrieval, retrieval complete, and retrieval abort.
3. Define the minimum status fields shown to the plugin: SD ready/open state, recording state, session/file path or id, generated samples, saved samples, queue drops, write errors, max write latency, overrun count, finalization reason, and local retrieval path.
4. Define reconnect semantics: host disconnect while recording starts a firmware-side grace timer; reconnect within the grace period resumes control of the same session; timeout finalizes and stops recording.
5. Define integrity checks for retrieved files: file size, chunk sequence/order, checksum or equivalent validation, and successful SD analyzer pass after local retrieval.
6. Update phase-local documentation with direct TCP and USB bridge expectations so implementers do not accidentally support one path only.

## Verification

- Protocol document names every state transition used by plugin, bridge, and firmware.
- Protocol explicitly prevents the plugin from reporting "saved" before firmware confirms finalization and retrieval.
- Protocol covers direct TCP and USB bridge mode.
- Protocol includes connection-loss and reconnect-before-timeout behavior.
- Protocol includes retry behavior for retrieval failures after the SD file is already finalized.
