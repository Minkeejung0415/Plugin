# Plan 06-01: Recording Protocol and Status Contract

**Phase:** 06 - Plugin-Controlled ESP32 SD Recording With Reconnect-Safe Local Retrieval
**Status:** Planned
**Wave:** 1
**Requirements:** SD-01, SD-03, SD-04, SD-05, LOSS-03, LOSS-04, STALL-03, OPS-01

## Scope

Define the command/status/file-transfer contract shared by the plugin, USB bridge, direct TCP path, and firmware before implementing transport-specific code. This plan must resolve the review feedback in `06-REVIEWS.md` before downstream code changes begin: transfer framing, live-stream separation, firmware/plugin/local state boundaries, protocol versioning, path safety, and analyzer-backed local verification are part of the contract, not implementation afterthoughts.

## Decision Trace

- D-01: The protocol must define board-side SD logging as the normal ESP32 recording path, not PC-side CSV capture.
- D-02: The protocol must make plugin RECORD the operator start/stop control for ESP32 SD recording.
- D-03: The protocol must prevent "Recording saved" or equivalent success until firmware finalization, retrieval, and local verification states justify it.
- D-04: The protocol must define stop/finalize followed by local retrieval of the completed SD file.
- D-05: The protocol must define local saving as post-record retrieval, not the live lossless acquisition path.
- D-06: The protocol must require analyzer-compatible retrieved files or a documented successor.
- D-07: The protocol must keep ESP32 SD logging active during host/plugin/bridge connection loss.
- D-08: The protocol must require a configurable reconnect grace constant near the top of supported firmware files.
- D-09: The protocol must preserve the same session when the host reconnects within the grace period.
- D-10: The protocol must require timeout finalization plus retained metadata for the next connection.
- D-11: The protocol must acknowledge record on/off, timeout finalization, and retrieval outcomes explicitly.
- D-12: The protocol must define the minimum status fields required by plugin and bridge surfaces.
- D-13: The protocol must require consistent direct TCP and USB bridge behavior for record, status, and transfer commands.

## Tasks

1. Document three separate lifecycle surfaces:
   - Firmware recording state: idle, starting, recording, host-disconnected-grace, finalizing, finalized, failed.
   - Transfer state: none, metadata, chunking, complete, aborted, failed.
   - Plugin/local state: command-sent, recording-confirmed, finalizing, sd-finalized, retrieving, local-copy-written, transfer-checksum-passed, analyzer-passed, analyzer-failed.
2. Define command names and responses for capability/version query, record start, status query, record stop/finalize, finalized-session metadata query, chunk retrieval by session id and offset, retrieval complete, retrieval abort, and error/status clearing.
3. Define concrete binary-safe retrieval framing: frame magic/version, frame type, session id, chunk index, byte offset, payload length, total file size, header CRC, payload checksum, EOF marker, abort marker, and retry/resume rules. The contract must define exact behavior for duplicate chunks, missing chunks, offset mismatch, checksum mismatch, and reconnect during transfer.
4. Define the command/status/file-transfer separation from live Open Ephys sample traffic. The contract must state whether transfer uses a command-only mode, a sidecar endpoint, or paused/isolation behavior, and must explicitly prevent file bytes from being interpreted as live sample frames.
5. Define the minimum status fields shown to the plugin: protocol version/capabilities, SD ready/open state, recording state, transfer state, session id, SD path token, generated samples, saved samples, queue drops, write errors, max write latency, overrun count, finalization reason, file byte size, file checksum, and local retrieval/analyzer result path.
6. Define reconnect semantics for each failure mode: direct TCP client disconnect, plugin-to-bridge disconnect, bridge process loss, USB/serial device loss, transfer disconnect, and firmware reboot. Host disconnect while recording starts a firmware-side grace timer; reconnect within the grace period resumes control of the same session; timeout finalizes and stops recording.
7. Define path and integrity safety rules: retrieval must use a firmware-issued session id or latest finalized session token, must not allow arbitrary filename reads, must be read-only against finalized SD files, and must never delete/truncate SD source files after local transfer.
8. Define integrity checks for retrieved files: file size, chunk order, whole-file checksum, transfer log, and successful SD analyzer pass after local retrieval. The contract must preserve the distinction between "SD finalized", "local transfer complete", "transfer checksum passed", and "analyzer continuity passed".
9. Update phase-local documentation with explicit direct TCP and USB bridge expectations so implementers do not accidentally support one path only.

## Verification

- Protocol document names every state transition used by plugin, bridge, and firmware.
- Protocol explicitly prevents the plugin from reporting "saved" before firmware confirms finalization and retrieval.
- Protocol covers direct TCP and USB bridge mode.
- Protocol includes connection-loss and reconnect-before-timeout behavior.
- Protocol includes retry behavior for retrieval failures after the SD file is already finalized.
- Protocol contains a concrete frame table with fields, error codes, timeout behavior, retry/resume rules, EOF behavior, and abort behavior.
- Protocol explicitly separates command/status/file-transfer traffic from live Open Ephys sample traffic.
- Protocol documents capability/version negotiation and unsupported-firmware behavior.
- Protocol forbids arbitrary SD path reads and limits retrieval to known finalized session ids/tokens.
- Protocol ties D-01 through D-13 from `06-CONTEXT.md` to named protocol decisions so downstream implementation can verify context coverage.
