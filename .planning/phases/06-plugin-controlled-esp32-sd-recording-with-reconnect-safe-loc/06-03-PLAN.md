# Plan 06-03: Plugin UI, Local Retrieval, and End-to-End Verification

**Phase:** 06 - Plugin-Controlled ESP32 SD Recording With Reconnect-Safe Local Retrieval
**Status:** Planned
**Wave:** 3
**Depends on:** 06-01, 06-02
**Requirements:** SD-05, LOSS-02, LOSS-03, LOSS-04, STALL-03, STALL-04, OPS-01, OPS-03

## Scope

Wire the Open Ephys plugin RECORD button to the new status/retrieval contract and verify that real operator runs save on ESP32 SD first, then retrieve a checked local copy. This plan must resolve the review feedback about asynchronous plugin threading, local output structure, timeout-finalized session UX, progress/cancel/retry behavior, unsupported firmware handling, and analyzer-backed evidence.

## Decision Trace

- D-01: Plugin behavior must treat ESP32 SD logging as the normal recording path and avoid silent PC-side CSV fallback.
- D-02: Plugin RECORD must be the operator start/stop control for ESP32 SD recording.
- D-03: Plugin UI must show firmware-confirmed and locally verified states instead of optimistic "saved" messages.
- D-04: Plugin stop must finalize the SD recording and retrieve the completed file to local storage.
- D-05: Plugin local saving must be represented as post-record retrieval, not live lossless capture.
- D-06: Plugin workflow must produce analyzer-compatible evidence for the retrieved local file.
- D-10: Plugin reconnect flow must surface timeout-finalized sessions retained by firmware.
- D-11: Plugin status handling must consume explicit acknowledgements for record on/off, timeout finalization, and retrieval outcomes.
- D-12: Plugin UI must expose or preserve the firmware status fields needed for truthful operator feedback.
- D-13: Plugin must support direct TCP and USB bridge behavior consistently or show clear unsupported-protocol status.

## Tasks

1. Update ESP32 plugin RECORD start behavior so it sends the agreed start command and waits for firmware status before showing recording confirmed.
2. Update RECORD stop behavior so it requests firmware finalization, waits for flushed/closed confirmation, then retrieves the finalized file to a local path.
3. Replace misleading status text such as "Recording saved" with state-specific messages: command sent, recording confirmed, SD failure, finalizing, finalized, retrieving, retrieved, retrieval failed, timeout finalized.
4. Implement finalize/retrieve work asynchronously. Network waits, firmware finalize polling, file transfer, checksum verification, and analyzer invocation must run off the Open Ephys audio/acquisition path and off the UI message thread; UI updates must be marshaled back safely.
5. Ensure the plugin no longer silently uses the legacy ESP32 PC-side CSV path for normal ESP32 SD recording. If firmware lacks the new protocol/capability response, show "recording protocol unsupported" or equivalent and do not claim SD recording.
6. Define and implement the local output structure for retrieved sessions: a timestamped/session-id directory containing the retrieved SD binary, metadata JSON, transfer log, checksum result, and analyzer summary.
7. Add safe retry behavior: if retrieval fails after SD finalization, preserve the SD file path/session id/token and allow the operator to retry retrieval without restarting acquisition or altering the SD source file.
8. Add progress, cancel, and retry UI/status states for long transfers, including serial/USB bridge mode where transfers may be slow.
9. Add timeout-finalized session discovery on reconnect/plugin restart. If firmware reports a preserved finalized session, the plugin must distinguish "SD finalized after disconnect" from "local copy verified" and offer retrieval of that session.
10. Add local retrieved-file verification by automatically invoking the existing SD analyzer when practical. If automatic invocation is not possible in the Open Ephys environment, write a machine-readable handoff file and exact command, then make the operator documentation treat analyzer pass/fail as required evidence.
11. Preserve the user-facing distinction between SD finalized, local transfer complete, transfer checksum passed, and analyzer continuity passed. Full success requires the analyzer-backed artifact.
12. Update operator documentation for the final workflow: insert SD, start plugin bridge if needed, press RECORD, stop RECORD, wait for retrieval, review transfer/analyzer results, and handle retry or timeout-finalized recovery.

## Verification

- No-SD dry run: plugin shows truthful SD failure instead of claiming a saved recording.
- SD smoke run: plugin starts SD recording, stop finalizes it, local retrieval completes, and firmware reports `sd_saved > 0` with `sd_errors = 0`.
- Retrieved file passes `esp32/host/analyze_sample_rate.py --format sd-bin`.
- Connection-loss run: disconnect during recording, reconnect before timeout, stop from plugin, retrieve file, and verify continuity.
- Timeout run: disconnect beyond grace period, reconnect, retrieve finalized file, and verify status says timeout-finalized.
- Open Ephys/plugin remains responsive during finalize/retrieval operations.
- Threading review confirms finalize/retrieve code does not block the audio/acquisition path or UI message thread.
- Plugin restart/reconnect test discovers a timeout-finalized preserved session and offers retrieval without claiming local verification first.
- Local retrieved-session directory contains binary data, metadata JSON, transfer log, checksum result, and analyzer summary or required machine-readable analyzer handoff.
- Unsupported or old firmware test shows a clear unsupported-protocol status and does not fall back to misleading legacy ESP32 CSV recording.
- Progress/cancel/retry behavior is exercised for a long or simulated-slow transfer.
- End-to-end evidence proves recording continued while the host was disconnected by analyzing SD continuity, not merely by checking status counters.
