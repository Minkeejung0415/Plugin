# Plan 04-01: Restore ESP32 14-Channel Magnetometer Contract Through Open Ephys

## Scope

Make the current 14-channel STEP firmware contract visible and consistent through the
Open Ephys plugin, USB bridge, and CSV/export path. The goal is to explain and fix why
magnetometer channels are not appearing in Open Ephys or the CSV saved from the plugin
without conflating this with acquisition sample loss.

This plan is intentionally narrow: it aligns channel counts, channel names, and conversion
paths. It does not change sensor fusion behavior, SD buffering, OpenSim IK, or unrelated
recording lifecycle code.

## Tasks

1. Confirm the live channel contract before editing.
   - Run or inspect firmware boot/handshake output and record the exact channel count.
   - Confirm whether the failing path is direct Wi-Fi TCP, USB bridge binary mode, USB bridge CSV mode, or SD retrieval.
   - Save the observed handshake lines in the execution summary.

2. Align plugin ESP32 fallback channel count with firmware.
   - In `Acqboardredpitaya.h`, update `ESP32_DEFAULT_CHANNELS` from 13 to 14.
   - Update the adjacent comment to include channel 13 as `dio`.
   - In `acqboard.ccp`, update stale comments that still describe ESP32 handshakes as 8 channels.
   - Do not change the dynamic handshake parser; if firmware or bridge reports a valid count, that count remains authoritative.

3. Verify Open Ephys channel naming for all 14 ESP32 channels.
   - Inspect `devicethread.cpp` and preserve the existing `esp32Names[]` order:
     `ax, ay, az, gx, gy, gz, mx, my, mz, qw, qx, qy, qz, dio`.
   - If any name/comment mismatch remains, correct comments only.
   - Do not rename channels in a way that breaks existing CSV or OpenSim tooling.

4. Align USB bridge binary/plugin handshake defaults.
   - In `esp32/host/serial_tcp_bridge.py`, change default `ESP32_NUM_CHANNELS` from 11 to 14.
   - Ensure plugin handshake replies advertise the configured `num_ch`.
   - Ensure binary frame normalization preserves the firmware header's `n_channels=14` and `num_bytes=28`.
   - Keep environment override support so older firmware can still be tested with `ESP32_NUM_CHANNELS=8`, `11`, or `13`.

5. Fix USB bridge CSV-mode packing so it does not drop magnetometer data.
   - Update `pack_csv_row()` to accept CSV rows with `seq` plus up to the configured channel count.
   - For current firmware, pack 14 int16 values and emit an Open Ephys header with `n_channels=14` and `num_bytes=28`.
   - If a CSV row has fewer channel values than the configured count, pad trailing channels with zero and log a warning or count.
   - If a CSV row has more values, ignore extras only after the configured count.
   - Preserve legacy 8-channel CSV behavior when `ESP32_NUM_CHANNELS=8`.

6. Align firmware banners and operator docs.
   - In `step_node.ino`, update serial text banners that list channel names but omit `dio`.
   - Update Open Ephys setup docs that still say current firmware uses 8 or 11 channels.
   - The docs should distinguish legacy channel counts from current 14-channel STEP firmware.

7. Add lightweight verification for bridge packing.
   - Add or extend a small host-side test/script check that feeds a synthetic 14-channel CSV row through `pack_csv_row()`.
   - Assert the packed header reports `n_channels=14`, `num_bytes=28`, `element_size=2`, and that channels 6-8 preserve `mx,my,mz`.
   - If there is no existing test harness for `serial_tcp_bridge.py`, add the smallest practical Python test under `tests/` or `esp32/host/`.

8. Bench verify in Open Ephys.
   - Start the same bridge/path used in the failing run.
   - Confirm the handshake says 14 channels.
   - Confirm Open Ephys shows `mx`, `my`, and `mz` channels.
   - Record a short session and export/retrieve CSV.
   - Confirm the CSV header contains `mx,my,mz`.
   - Move the IMU or bring a magnet nearby and confirm magnetometer values change.

## Verification

Automated:

- `python -m pytest` or the repo's existing focused test command passes for any added bridge test.
- Synthetic CSV bridge test proves `mx,my,mz` survive at channels 6-8.
- Static search shows active STEP defaults agree on 14:
  - `NUM_CHANNELS 14`
  - `ESP32_DEFAULT_CHANNELS = 14`
  - `DEFAULT_NUM_CHANNELS` fallback `"14"`
- Static search confirms current channel-order docs include `dio` after `qz`.

Manual/bench:

- Bridge/direct handshake advertises `14 channels`.
- Open Ephys channel list includes `mx`, `my`, and `mz`.
- Saved/exported CSV includes `mx,my,mz`.
- Magnetometer values are non-constant when the magnetic field changes.

## Success Criteria

- The plugin exposes all 14 ESP32 channels for current STEP firmware.
- `mx`, `my`, and `mz` are visible in Open Ephys.
- USB bridge binary mode and CSV mode both preserve magnetometer channels.
- The CSV saved/exported from the plugin includes magnetometer columns or values.
- Docs and boot/status text no longer tell operators to expect 8, 11, or 13 channels for the current 14-channel firmware unless explicitly labeled legacy.
- Any remaining absence of magnetometer data is classified with evidence as one of:
  firmware `mag_ok=false`, bridge path mismatch, Open Ephys export setting, or downstream display/export issue.
