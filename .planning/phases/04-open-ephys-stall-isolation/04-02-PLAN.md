# Plan 04-02: Isolate Lossy UDP Streaming From Lossless SD Acquisition

## Objective

Move Open Ephys sample delivery out of the acquisition loop and onto a bounded, lossy
UDP task. Preserve the approximately 1 kHz filter+SD envelope as closely as practical.
SD continuity is ground truth; UDP may drop under congestion. TCP remains available for
commands, status, rec-v1 finalization, and file retrieval.

## Tasks

### 1. Define the firmware stream queue and counters

**Files:** esp32/arduino/step_node/step_node.ino

**Action:**
- Add a fixed stream record containing the Open Ephys header, 14-channel payload,
  sequence number, and acquisition timestamp.
- Add a small bounded queue optimized for freshness, not lossless buffering.
- When full, discard the oldest queued stream record, offer the newest without waiting,
  and increment stream_queue_drops.
- Track offered, enqueued, sent, queue drops, send errors, maximum depth, and maximum
  UDP send duration.
- Reset stream counters at START and expose them in STATUS separately from SD counters.
- Never include stream failures in sdErrorTotal or recording success.

**Verify:**
- Static review proves stream enqueue cannot wait.
- STATUS labels SD and UDP counters unambiguously.
- Focused tests cover clean SD with stream drops.

**Acceptance:**
- Congestion changes only stream counters and cannot delay acquisition or alter SD errors.

### 2. Create the UDP task and make acquisition SD-first

**Files:** esp32/arduino/step_node/step_node.ino

**Action:**
- Add WiFiUDP and create a sender task pinned to core 0 below sdWriteTask priority 4.
- Send complete frames to the active TCP controller remoteIP on UDP port 55001.
- Disable TCP sample writes in the new profile; preserve TCP commands, rec-v1, and retrieval.
- Reorder acquisition so logSd executes before one non-blocking stream offer.
- Remove TCP and serial sample writes from the timed acquisition path. A retained bench
  transport must use a background queue or be excluded from the production profile.
- Ensure STOP, disconnect, absent receiver, and UDP failure never stop SD recording.
- Print acquisition, SD, and UDP core IDs and priorities once at boot.

**Verify:**
- arduino-cli compile --fqbn esp32:esp32:XIAO_ESP32S3 esp32/arduino/step_node
- Static search finds no direct TCP, serial, or UDP write in the acquisition loop.
- Boot output proves acquisition core 1; SD and UDP core 0; SD priority above UDP.

**Acceptance:**
- Acquisition contains sensor/filter/packing plus two non-blocking queue offers only.
- A missing receiver cannot prevent SD enqueue or finalization.

### 3. Route ESP32 samples through UDP in the plugin

**Files:** acqboard.ccp, Acqboardredpitaya.h

**Action:**
- Add an explicit ESP32 transport capability/mode; never infer UDP by misclassifying the
  ESP32 as Red Pitaya.
- Keep the command socket dedicated to control and rec-v1 responses.
- For UDP-capable firmware, bind port 55001 and parse the same 22-byte header and payload.
- Preserve the 14-channel order ax,ay,az,gx,gy,gz,mx,my,mz,qw,qx,qy,qz,dio and scaling.
- Validate each datagram independently; discard malformed or wrong-size packets without
  desynchronizing subsequent packets.
- Preserve legacy TCP-sample compatibility or show a clear unsupported transport state.
- Ensure stop unblocks UDP receive without racing TCP recording finalization/retrieval.

**Verify:**
- Plugin build succeeds for ESP32 UDP and legacy TCP branches.
- Parser tests cover valid, truncated, oversized, and wrong-channel datagrams.
- TCP control and rec-v1 remain functional while UDP samples flow.

**Acceptance:**
- Open Ephys consumes available UDP frames without owning or blocking SD.
- Packet loss creates display gaps only; the next valid datagram remains usable.

### 4. Separate SD verdicts from stream observations

**Files:** esp32/host/stress_test_serial.py, tests/test_esp32_stream_isolation.py

**Action:**
- Parse new stream counters from STATUS.
- Report requested rate, generated, SD saved/errors/drops, UDP offered/sent/drops/errors,
  SD-derived rate, and host-visible rate.
- Split results into sd_pass and stream_observation. Integrity passes when finalized SD
  and counters pass even if UDP drops packets.
- Retain an optional strict transport verdict for Open Ephys quality diagnosis.
- Test that UDP drops do not fail clean SD, while any SD gap, drop, error, or
  generated/saved mismatch fails.
- Preserve stream-off and historical result parsing.

**Verify:**
- python tests/test_esp32_stream_isolation.py
- Existing stress and bridge tests pass.
- Synthetic stream drops with perfect SD yield sd_pass=true and degraded streaming.

**Acceptance:**
- Reports cannot confuse lossy visualization with lossless persistence evidence.

### 5. Document and bench-verify the isolation envelope

**Files:** esp32/docs/open-ephys-plugin.md, esp32/docs/stress-test-sample-rate.md

**Action:**
- Document TCP control plus UDP 55001 samples, task priorities, queue policy, firewall
  needs, and SD-first semantics.
- Preserve the 2026-06-18 stream-off baseline: 950 requested -> 944.5 measured PASS;
  1000 -> 975.8 PASS; 1050+ flattens near 992.
- Run 60-second filter+SD tests at 950 and 1000 Hz with UDP/Open Ephys enabled.
- Run a receiver-stall test by pausing, disconnecting, or slowing the UDP consumer.
- Finalize and analyze SD files for duplicates, missing sequences, timestamp gaps,
  duration/rate mismatch, and generated/saved agreement.
- Record UDP drops separately and never claim lossless streaming.

**Verify:**
- Normal receiver: SD has zero gaps, duplicates, drops, and errors and reaches at least
  95% of requested rate at the selected operating point.
- Stalled receiver: UDP drops may rise, but SD remains perfect and reaches at least 97%
  of its corresponding stream-off baseline.
- Open Ephys resumes from later valid datagrams.
- Documentation states the tested safe cap and residual Wi-Fi, CPU, and card dependence.

**Acceptance:**
- Evidence classifies Open Ephys degradation as downstream when SD stays continuous.
- Production rate is selected from finalized SD analysis with safety headroom.

## Verification

Automated:
- Arduino firmware compiles for XIAO ESP32S3.
- Plugin build succeeds.
- Stream-isolation and existing bridge tests pass.
- Acquisition loop contains no direct transport writes.
- Synthetic verdict tests preserve SD pass when only UDP drops occur.

Hardware:
1. Establish stream-off filter+SD baselines at 950 and 1000 Hz.
2. Repeat with Open Ephys UDP active.
3. Repeat while UDP receiver is stalled or disconnected.
4. Finalize and analyze each SD log.
5. Compare SD continuity/rate and stream counters across runs.

## Success Criteria

- SD and UDP writers are separate core-0 tasks with SD priority higher than UDP.
- Acquisition stays on core 1 and performs no blocking transport write.
- SD enqueue precedes the best-effort stream offer for every generated sample.
- TCP is control/retrieval only for the new profile; samples use UDP 55001.
- Stream saturation increments UDP counters without SD drops, errors, gaps, or
  generated/saved disagreement.
- Filter+SD+UDP preserves perfect SD continuity and remains within 3% of stream-off SD
  rate at the selected production frequency.
- Receiver-stall testing proves Open Ephys loss does not corrupt SD acquisition.
- Documentation identifies SD as ground truth and UDP as a lossy live view.
